## Results and Discussion

A lightweight LogBERT-style prototype was developed and tested on the HDFS v2 log dataset. The preprocessing pipeline first converted the raw HDFS logs into structured log records, then generated event templates and event IDs, and finally grouped logs into block-based sessions using `BlockId`. To make the workflow manageable on limited system memory, a reduced restart-safe dataset was created and saved for later stages. This reduced dataset contained 39,998 training sessions and 10,000 test sessions, with each session represented as a fixed-length sequence of 20 event tokens. A reduced vocabulary of 50,003 tokens was also created to keep the model small enough to train without crashing.

The final model was a lightweight Transformer-based LogBERT-style network trained on CPU. Training progressed successfully over two epochs. In the first epoch, the average training loss gradually decreased and finished at 3.1549. In the second epoch, the loss decreased further and finished at 2.5700. This reduction in loss shows that the model was able to learn useful sequential patterns from the log sessions. The average test loss was 0.3874, which suggests that the trained model generalized reasonably well to the held-out test data under the reduced prototype setting.

After training, anomaly scores were computed for the test sessions. The anomaly score for each session was based on masked-token prediction difficulty, where a higher score indicated that the model found the session more unusual. The scored sessions were ranked from most suspicious to least suspicious. The highest anomaly scores were around 15, and the top anomalous sessions were linked back to their corresponding HDFS block IDs and session lengths. For example, the most anomalous session had block ID `blk_1073828024`, sequence length 21, and anomaly score 15.533759. Other highly ranked anomalous sessions also had substantial sequence lengths, including one session with length 50, indicating that longer and more complex sequences may contribute to abnormal behavior patterns.

To make the output interpretable, the event ID sequences were decoded back into readable event templates. This showed that many of the top-ranked anomalous sessions were associated with receiving and block-transfer related HDFS operations. This is meaningful because abnormal block receiving, transfer, or replication behavior can indicate instability, inconsistency, or unusual data node activity in a distributed file system. The readable anomaly outputs make the prototype more useful for manual inspection and future dashboard integration, because the flagged sessions are no longer only numeric scores but can also be inspected as understandable log patterns.

Overall, the prototype demonstrates that a LogBERT-style anomaly detection workflow can be implemented successfully on HDFS logs even under hardware and memory limitations by using reduced datasets and restart-safe saved states. The system was able to preprocess logs, train a Transformer-based sequence model, score test sessions, and identify the most anomalous block sessions in a human-readable form. While this version is a lightweight approximation rather than a full-scale research reproduction of LogBERT, it provides a strong proof of concept for sequence-based log anomaly detection and offers a practical base for future improvement.

## Limitations

This prototype used a reduced subset of the full HDFS v2 dataset in order to keep computation feasible on CPU and avoid repeated kernel crashes. Because of that, the model was not trained on the full distribution of available sessions. The event template generation process was also simplified and relied on regex-based normalization rather than a dedicated industrial-strength log parser such as Drain. As a result, some event patterns may still be more fine-grained than ideal. In addition, anomaly scoring was based on reconstruction difficulty rather than comparison with a fully labelled anomaly benchmark, so the ranked anomalous sessions should be interpreted as strong anomaly candidates rather than confirmed malicious or faulty events.

## Future Improvements

Future work could improve the system by training on a larger portion of the HDFS dataset with better hardware support, especially GPU acceleration. Event parsing could also be improved using a stronger log parsing framework to reduce noise in the template vocabulary. Another useful extension would be to connect the anomaly scores and readable sessions into a small dashboard or monitoring view, so the flagged sessions can be explored more easily. Finally, evaluation could be strengthened by using labelled HDFS traces or benchmark anomaly labels, allowing the model’s anomaly detection quality to be measured with precision, recall, and F1-score.
